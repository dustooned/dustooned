# Asset Guide

Place production assets under `public/images/`.

Recommended folders:

```txt
public/images/motion/{slug}/
public/images/illustration/{slug}/
public/images/interactive/{slug}/
public/images/journal/{slug}/
public/logos/
public/og/
public/icons/
```

Recommended dimensions:

- Project thumbnail: 1600 x 1200, 4:3
- Journal thumbnail: 1600 x 1200, 4:3
- Motion poster: 1920 x 1080, 16:9
- Project hero: 2400 x 1350, 16:9
- Portrait illustration: 1600 x 2000, 4:5
- Landscape illustration: 2000 x 1500, 4:3
- Open Graph: 1200 x 630

Use lowercase kebab-case file names.

## Thumbnails: use the generator, not manual exports

`thumb.webp` files (project + journal grid cards) and each journal post's `image` (hero)
field are generated by:

```sh
node scripts/generate-smart-thumbnails.mjs
```

It reads `hero.*` (illustration) or `image-1.*` (journal) from each asset folder, smart-crops
to 1600x1200, and writes `thumb.webp` next to it — then rewrites the `thumbnail:` (and for
journal, `image:`) frontmatter fields automatically. Re-run it any time you drop in a new
`hero`/`image-1` source file rather than hand-exporting a `thumb.webp` yourself, or you'll
bypass the animated-GIF handling below and any future crop/quality tuning.

### Gotcha: animated GIF sources must stay animated, but small

If the source file is a `.gif`, the script generates an **animated** `thumb.webp`
(`sharp(src, { animated: true })`) instead of freezing it to one frame — but at a much
smaller size than static thumbnails: `GIF_THUMB_W/H = 640x480` at quality 65, vs. 1600x1200 /
quality 82 for static sources. This split exists because animated-webp cost scales with pixel
count *per frame*; the first version of this fix generated GIF thumbnails at the full
1600x1200 static size, which produced multi-megabyte files (up to ~10MB for a ~24-frame gif)
and visibly lagged the `/journal` and Illustration grids. Do not raise `GIF_THUMB_W/H` back up
without re-checking output file sizes — grid cards only ever render a few hundred px wide, so
640x480 is already 2x+ retina headroom.

Do not reintroduce `{ animated: false }` for the GIF branch either — that's the earlier,
separate bug (frozen thumbnails). Both constraints (animated *and* small) matter.

Sharp's animated-GIF resize does not support `position: sharp.strategy.attention` (that's a
single-frame feature); the GIF branch uses `position: "centre"` instead. Keep that split when
editing the script.

### Gotcha: journal `thumbnail` (grid) vs. `image` (post hero) are different assets

`JournalLayout.astro` does **not** reuse the cropped grid `thumbnail` as the post's hero
image. It renders `data.image` (the original, unresized/uncropped `image-1.*` source file)
and only falls back to `thumbnail` if `image` isn't set (e.g. video-embed-only posts with a
YouTube thumbnail and no local image). The generator script writes both fields for any post
with a local `image-1.*` source — see `docs/JOURNAL_SYSTEM.md`. If you ever see a journal
post's hero looking cropped/zoomed like its grid card, check whether `image` is missing from
its frontmatter.

### Gotcha: portrait/cover hero images and desktop width

Illustration entries with `heroAspect: "auto"` (book covers, posters — no fixed aspect-ratio
crop) render at their natural aspect ratio. Without a cap, a tall portrait cover rendered at
`width: 100%` of the ~1040px wide-shell container could blow up to 1500px+ tall on desktop.
`src/styles/layout.css` caps this specifically for the no-`data-aspect` case:

```css
.project-hero__image:not([data-aspect]) {
  max-width: min(100%, 480px);
  max-height: 70vh;
  object-fit: contain;
  margin-inline: auto;
}
```

This only applies when `heroAspect` is `"auto"`. Fixed-aspect heroes (`16:9`, `4:3`, `4:5`,
`1:1`) are unaffected since `aspect-ratio` + `object-fit: cover` already bounds their height.

## Logo

Logo files live in `public/logos/`:

- `logo.svg` — desktop nav mark, referenced in `SiteHeader.astro`
- `logo-wordmark.svg` — mobile nav wordmark
- `logo-mark.svg` — spare/alternate mark, not currently wired into a component

Nav CSS already auto-sizes whatever is dropped in (`height: 44px; width: auto;` desktop,
`height: 38px; width: auto;` mobile in `src/styles/nav.css`), so you do not need to
pre-resize a replacement logo — just overwrite the file in place. Prefer SVG (scales
losslessly at any size); if using a raster export, target roughly 2x the display height
(~100–150px tall) for retina sharpness. Verify a fresh SVG export doesn't contain a raster
`<image>`/base64 payload disguised as an SVG (common from some export tools) — that would
defeat the point of using a vector format.
